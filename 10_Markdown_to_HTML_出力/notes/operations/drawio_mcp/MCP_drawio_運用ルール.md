# draw.io MCP 運用ルール（改訂版）

このファイルは、draw.io MCP で業務フロー図を作成するときの標準手順です。  
以後、作業前に必ず本ファイルを読み、内容に従って対応すること。

## 1. 対象データの見直し結果（notes/operations/drawio_mcp/sample_data.txt）
- 元データは「講座運営全体フロー + システムI/O一覧」。
- 人物レーンで強調すべき登場主体は `受講生` と `TA`。
- 主要処理の大半は `運営システム` 側で実行される。
- したがって図は `受講生 / TA / 運営システム` の3レーンを基本構成とする。

## 2. データ解釈ルール（雑化防止）
- 工程名は意味を保ったまま短縮する（省略しすぎない）。
- 分岐は最低限以下を明示する。
  - 取込エラー（Yes/No）
  - 満室判定（Yes/No）
- `notes/operations/drawio_mcp/sample_data.txt` の固有ルールで重要なものは削らない。
  - 先着順判定
  - 外部宿泊フラグ
  - カード紛失時の抹消+再登録
  - SAP連携
- 「図の見た目を優先して意味が欠落」する変更は禁止。

## 3. レイアウト固定要件
- フロー方向は必ず縦: `flowchart TB`
- レーンは横並び3列（左から `受講生` `TA` `運営システム`）。
- 各レーンは「大きい長方形コンテナ + 上部小ラベル」で表現する。
- レーン同士の上端/下端を揃える（トップ/ボトム一致）。
- 中央にレーン名だけ出る表示は使わない。

## 4. 生成ポリシー
- まず最小構文で安定表示版を作る。
- 表示確認後に装飾（色/枠）を追加する。
- エラー時は即フォールバックして再生成する。
  - ノード文言の簡素化
  - 分岐ラベル簡素化
  - スタイル削減

## 4.1 横化の原因と防止パターン（今回の知見）
- 原因:
  - `flowchart TB` でも、外側ラッパーで `direction LR` を使うと全体が横型に見えやすい。
  - レーン間の左右矢印が多すぎると、体感レイアウトが横寄りになる。
- 防止パターン:
  - `ROW direction LR` のような外側LRラッパーは使わない。
  - レーンは個別 `subgraph` で定義し、各レーン内は `direction TB` を維持。
  - 上段ラベル同士を `SH --- TH --- OH` で同列固定する。
  - レーン間矢印は最小限にし、主経路は縦方向で読めるように配線する。
  - 「文字の縦書き」と「フロー縦方向」は別概念として扱う。

## 4.2 線スタイル運用（2段階）
- 箱間の線スタイルは、次の2段階で整える。
- `Mermaid生成段階`
- 薄い灰色の線色は `linkStyle` で指定する。
- 点線は `-.->` を使って指定する。
- 直角線（elbow）や全リンクの完全統一は、変換時に崩れる場合がある。
- `draw.io仕上げ段階`
- draw.ioで線を一括選択し、色・点線・直角線を最終統一する。
- 見た目品質の最終担保はこの段階で行う。
- `代替案（安定重視）`
- Mermaidでは構造（ノードと接続）を優先して作る。
- 線デザインはdraw.io側テンプレで統一する。

## 4.3 ユーザー修正版XMLからの学習知見（FB.drawio.xml）
- 参照: `notes/operations/drawio_mcp/FB.drawio.xml`
- 今回の確定意図:
- 3レーン（`運営システム` `TA` `受講生`）を縦長コンテナで明確に分離する。
- 各レーンの最上部に、見出し用の小ラベル長方形を置く。
- 全体は上から下へ読む時系列を優先し、縦方向の情報密度を許容する。
- ノードは全体で同一高さに統一し、視認性を保つ。
- 分岐点（例: `取込エラー`, `満室判定`）はダイヤ形を維持する。
- Yes/Noラベルは省略しない。
- 線は装飾より可読性優先（モノトーン基調）で運用する。
- 備考:
- FB.drawio.xml は曲線エッジ（`curved=1`）が多い。
- 直角線を必須化したい場合は、別途「直角線優先」ルールを追加して運用する。

## 5. 提出前セルフレビュー（毎回必須）
- URL提示前に必ず自己レビューする。
- 以下がすべて PASS であること。
  - `flowchart TB` になっている
  - 外側 `direction LR` ラッパーを使っていない
  - 3レーン横並びになっている
  - 上部小ラベル + 大コンテナ構成になっている
  - 全ノードの高さが統一されている
  - レーン上下高さが揃っている
  - 文言が雑に省略されていない
  - Yes/No 分岐が読める
- 1つでも FAIL なら再生成し、合格版のみ提示する。

## 6. レビュー結果の提示フォーマット
- 毎回、図URLと一緒に3行で報告する。
- 形式:
- `Self-Review: PASS/FAIL`
- `Checked: TB, lanes, label-box, alignment, readability, branch`
- `Fix applied: <必要時のみ>`
- `Geometry: ノード高さ方針, Y開始位置方針`
- 可能な場合は、代表ノードの高さとY開始座標を実数値で併記する。

## 7. 変更依頼時の扱い
- ユーザー指示を `再生成` か `既存修正` か明示する。
- 微調整が連続する場合は XML ベース編集を提案する。

## 8. 応答ルール
- MCP実行前に、今回の反映ポイントを1-3行で宣言する。
- 実行後は「何を変えたか」を短く列挙する。
- エラー時は原因候補と次対処をセットで示す。
- 図を渡すときは、各レーンのノード高さの扱いと、Y開始位置の扱いを必ず報告する。

## 9. 役割分担（誰が何をするか）
- ユーザーが担当:
- レーン外枠・ラベル枠の標準フォーマット決定（寸法、位置、見た目）
- 最終デザイン判断（線種、余白、装飾の最終承認）
- 必要に応じたXMLの手修正と、意図の指示
- Codex（私）が担当:
- 業務データから工程抽出、工程のレーン振り分け（受講生/TA/運営）
- フロー接続、分岐構造（Yes/No）、文言粒度の整合
- Mermaid下書き生成、必要に応じたXML編集提案
- 修正版XMLの差分読解と、運用ルールへの反映

## 10. 実務フロー（1サイクル）
- 1. ユーザーが要件と入力データ（必要ならXML）を渡す。
- 2. CodexがMermaidで下書き図を作成し、レビュー結果を添えて提示する。
- 3. ユーザーが見た目を調整（必要ならXMLで寸法・配置を固定）。
- 4. ユーザーが修正版XMLと意図を渡す。
- 5. Codexが修正意図を抽出し、本MDへルールとして追記する。
- 6. 次回生成で更新済みルールを適用して再現性を上げる。

## 11. 受け渡し時の最低入力
- `業務内容`: もとになるテキスト（例: sample_data.txt）
- `優先事項`: 何を固定したいか（例: 外枠寸法、線種、高さ統一）
- `修正時`: 修正版XML + 変更意図（箇条書きで可）
